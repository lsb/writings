SIMD Urban API.

Call all bookstores in a country.

Problem—Methods—Quality Control—APIs—Results

Problem

If you have a book in print, like a friend of mine does, how do you tell in an hour or so what bookstores have it in stock?

Methods

The phone is the best API to the real world.

Calling separate bookstores is a trivially parallelizable task. People need to be hired for a short duration to make calls. Instead of having a human hire workers (like a butler or TaskRabbit), offer work on the elastic labor pool of Mechanical Turk. Everyone has cheap access to a phone or to Skype.

Quality Control

The standard way of measuring whether people do rote work well is to include questions with known answers, set an error rate threshold, and use the threshold to either keep the work or to get another response.

So we need to give workers numbers of bookstores with known availability/unavailability.

Instead of using a physical bookstore, which might hang up on workers after the tenth call, we can make several “bookstores” via Twilio, and pre-program their responses.

APIs

The easiest way of abstracting over the low-level details of Mechanical Turk is to use the REST api of Ask–Human that simplifies the operation of asking a question. Issue a PUT to pose a question, and poll for an answer (as a JSON array) with a GET.

The nicest API for dealing with city data is Foursquare's API. We can find all the zip codes in the US, do a 4sq search by zipcode, a refinement to only venues it considers bookstores, keep all the ones with phone numbers, and aggregate them into a list of all the bookstores in a city/region/state.

Using Ask–Human it’s under a dozen lines of code to perform the full query, assuming you have Twilio numbers set up.

Thirty bucks later, and, out of 39 bookstores near SF, 7 have it in stock, 24 are out of stock, 6 phone numbers were bad, and 2 phone numbers did not correspond to bookstores.
